# Pola Requirement Analyzer

## 目标

把模糊输入转成可验证、可设计、可编码的需求。尤其适合从手写图、会议记录、用户描述和项目文件中提炼开发任务。

## 需求质量标准

一条需求进入架构或编码前，至少要具备：

- **目标用户**：谁会使用或受影响。
- **触发场景**：用户在什么情况下需要它。
- **输入**：用户、系统或外部服务提供什么。
- **输出**：界面、文件、接口响应、任务结果或状态变化是什么。
- **验收标准**：可以通过命令、页面、接口、日志或人工检查验证。
- **非目标**：本次明确不做什么。
- **风险**：权限、数据、计费、部署、AI 不确定性、第三方 API 等。

## 输入来源

- 用户自然语言描述
- 图片或手写流程图
- 现有 README、开发日志、代码目录
- issue、需求池、PRD、聊天记录
- 竞品或官方文档链接

## 默认流程

1. **提取原始信息**
   - 从文字和图片中列出明确出现的阶段、动作、角色和产物。
   - 不确定内容标记为推断。

2. **形成需求口径**
   - 写清用户目标、目标用户、触发场景、输入物、输出物。
   - 写清非目标，避免需求无限扩张。

3. **拆验收标准**
   - 每条验收标准必须可验证。
   - 标注验证类型：文档、代码、单测、集成、UI、部署、回归、日志、git。

4. **风险与澄清**
   - 涉及生产、权限、数据迁移、计费、第三方 API、AI 不确定输出时必须列风险。
   - 如果风险会改变实现方向，先让用户确认。

5. **任务拆解**
   - 输出按阶段执行的任务列表。
   - 每个任务绑定一个验收项或产物。

## 需求分类

根据内容标注类型，影响后续验证策略：

- `新功能`：需要完整验收、测试和发布说明。
- `Bug 修复`：必须保留原始复现 case 和回归验证。
- `体验优化`：需要 UI 或交互验证，必要时截图。
- `技术债`：需要证明无行为回归，重点看测试和风险。
- `配置/运营`：需要环境、权限、回滚和观察项。
- `文档/skill`：需要结构校验和示例覆盖。

## 澄清问题规则

只问会改变方案或验收的问题。优先问：

- 输出物到底是什么，格式和保存位置是什么。
- 哪些用户路径必须真实验证。
- 是否允许改动数据结构、接口或部署配置。
- 是否必须兼容旧数据、旧 API、旧页面。
- 上线范围是本地、测试、灰度还是生产。

不要为了追求完美 PRD 一次问一长串问题；能合理假设的，写入“假设”并继续规划。

## 验收标准写法

每条验收标准使用 `A编号`：

```markdown
- A1 文档：生成需求分析和架构开发文档，包含范围、非目标、模块影响、测试策略。
- A2 代码：新增/修改的功能在目标入口可被调用，旧路径保持兼容。
- A3 单测：新增逻辑有最小单元测试，命令退出码为 0。
- A4 集成：真实用户路径可完成，记录 URL/任务 ID/截图/日志。
- A5 发布：发布清单包含部署面、回滚点和发布后验证。
```

验收标准禁止写成“体验更好”“性能更优”这类不可验证表述，除非补上具体指标。

## 图片处理规则

- 先描述图片中可读内容，再给推断。
- 对手写不清楚的词用“疑似”标记。
- 不把图片中的草稿直接当最终需求，要整理成结构化口径。
- 如果图片和用户文字冲突，以用户最新文字为准，并说明差异。
- 图片里出现的箭头、编号、括号、勾选要转成流程阶段或依赖关系。
- 图片内容不清时保留“未确认项”，不要强行脑补。
