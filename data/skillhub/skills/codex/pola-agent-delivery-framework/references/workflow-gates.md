# Pola 交付门禁

## Phase 0 项目上下文

输出：
- 技术栈
- 启动命令
- 测试命令
- 主要目录
- 部署方式
- 已有规范文件

阻塞：
- 项目无法读取
- 缺少关键依赖且无法判断命令

通过条件：
- 已明确项目类型和主要技术栈。
- 已找到或确认缺失测试命令。
- 已确认是否为 git 仓库。
- 已列出后续阶段必须遵守的项目规范。

## Phase 1 需求

输出：
- 需求目标
- 用户路径
- 非目标
- 验收标准
- 风险和待澄清项

阻塞：
- 验收标准不可验证
- 用户目标与输出物不明确

通过条件：
- 每个核心目标至少有一个验收项。
- 非目标和假设已写明。
- 高风险项已进入澄清或风险列表。

## Phase 2 架构

输出：
- 影响模块
- 数据流
- 接口和数据结构
- 文件改动计划
- 测试策略
- 部署和回滚计划

阻塞：
- 涉及数据迁移、计费、生产环境或第三方 API，但缺少约束信息

通过条件：
- 文件改动计划清晰。
- 测试策略覆盖核心验收。
- 部署和回滚影响已判断。

## Phase 3 编码

输出：
- 代码 diff
- 关键决策
- 与验收项的映射

阻塞：
- 相关文件有冲突性未提交改动
- 需要密钥、账号或外部资源

通过条件：
- diff 对应架构计划。
- 没有无关大改。
- 新增或修改测试已同步考虑。

## Phase 4 Review

输出：
- 发现的问题
- 修复项
- 残余风险

阻塞：
- P0/P1 问题未修复且未被用户接受

通过条件：
- P0/P1 为 0。
- P2/P3 有处理结论。
- 测试缺口进入 test gate。

## Phase 5 测试

输出：
- 命令
- 结果
- 日志位置
- 未覆盖原因

阻塞：
- 核心路径测试失败

通过条件：
- 最小相关测试通过。
- 未运行项有明确原因。
- 失败不被模糊归为“环境问题”。

## Phase 6 发布

输出：
- 发布清单
- 环境
- 命令
- 回滚点
- 发布后验证项

阻塞：
- 生产高危动作未确认
- 没有回滚方案

通过条件：
- 发布面明确。
- 回滚点明确。
- 发布后回归项明确。

## Phase 7 收尾

输出：
- 开发日志
- commit
- push 状态
- 需求状态回填

阻塞：
- diff 中有秘密、临时代码或无关大改

通过条件：
- 文档和 diff 一致。
- git 状态清楚。
- commit/push 状态如实记录。
