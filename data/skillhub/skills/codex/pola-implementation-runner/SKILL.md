# Pola Implementation Runner

## 目标

把已确认的需求和架构方案落到代码中，同时保持改动范围可控、风格贴合项目、每个改动可验证。

## 输入

开始编码前应掌握：

- 需求口径和验收标准。
- 架构开发文档或轻量实现计划。
- 项目画像中的技术栈、命令和约束。
- 当前工作区状态。
- 相关文件和相似实现。

如果缺少其中任一项，先补齐或记录合理假设。

## 开工前检查

1. 已有需求口径和验收标准。
2. 已有架构开发文档或足够明确的小需求说明。
3. 已读取相关代码和项目约定。
4. 已检查工作区状态，理解相关脏文件。

## 代码阅读方法

1. 用 `rg` 搜索入口、路由、组件、服务、测试和配置。
2. 找一个最相近的已实现功能作为风格参考。
3. 读调用链上下游，至少覆盖：
   - 用户入口或 API 入口。
   - 核心业务逻辑。
   - 数据持久化或外部调用。
   - 错误处理和日志。
   - 现有测试。
4. 如果项目有规范文件，先遵守项目规范，不发明新模式。

## 编码流程

1. **定位现有模式**
   - 搜索相似功能、测试、配置、错误处理、日志方式。
   - 优先复用现有 helper、组件、服务和脚本。

2. **小步实现**
   - 先改核心路径，再补边界处理。
   - 只改与需求相关的文件。
   - 不做无关格式化和重构。

3. **同步测试**
   - 能补单测就补最小单测。
   - 涉及共享逻辑时扩大测试覆盖。

4. **记录决策**
   - 对非显然选择，在最终回复或架构文档中记录原因。

## 改动粒度

- 单文件小修：直接修改并补最小验证。
- 多文件功能：按“数据结构 -> 业务逻辑 -> 入口 -> UI -> 测试”顺序推进。
- 跨模块改动：每完成一个模块就运行可行的局部检查。
- 数据或配置改动：先写迁移/回滚说明，再改代码。

## 测试同步策略

- 纯函数或解析逻辑：补单元测试。
- API 行为：补 handler/service 测试或集成测试。
- UI 行为：补组件测试或浏览器验证脚本。
- bugfix：补会在旧代码失败的新测试，或记录无法自动化的复现步骤。
- skill 文档：运行 skill 校验脚本和脚本语法检查。

## 变更映射表

编码过程中维护：

```markdown
| 验收项 | 文件 | 改动 | 验证 |
| --- | --- | --- | --- |
| A1 | src/foo.ts | 新增解析逻辑 | unit test |
```

## 遇到脏文件

- 先用 `git status --short` 和 `git diff -- <file>` 理解已有改动。
- 如果已有改动与本任务无关，不触碰。
- 如果已有改动与本任务相关，在其基础上继续，不回滚。
- 如果无法判断是否安全，暂停并说明冲突点。

## 禁止项

- 不覆盖用户未提交改动。
- 不提交密钥、token、个人路径。
- 不把临时代码、调试输出、硬编码测试账号留在代码里。
- 不在未验证时声称完成。
- 不用大规模格式化掩盖实际 diff。
- 不为了通过测试降低业务校验或删除断言。

## 输出

```markdown
**实现摘要**
- 改动文件：
- 验收项映射：
- 新增测试：
- 风险：
```

## 完成标准

- 代码实现覆盖核心验收标准。
- 有最小相关测试或明确无法自动化原因。
- diff 中没有明显临时代码、密钥和无关格式化。
- 后续 review 和 test gate 可以接着执行。
