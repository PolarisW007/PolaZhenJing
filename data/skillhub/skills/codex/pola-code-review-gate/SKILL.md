# Pola Code Review Gate

## 目标

在测试和发布前，用审查视角找出会影响用户、数据、稳定性、安全和部署的真实问题。

## 输入

- 需求口径和验收标准。
- 架构开发文档。
- 当前 diff 或指定文件。
- 已运行测试结果。
- 部署或运行环境说明。

如果只给代码没有需求，先从代码行为推断预期，并在输出中标记假设。

## Review 顺序

1. 读取需求和架构计划，明确预期行为。
2. 查看 diff 和相关上下文。
3. 按严重程度列发现项。
4. 对可直接修复的问题，修复后重新 review。
5. 对无法修复的问题，记录 blocker 或用户需确认的风险。

## 检查项

- 行为是否覆盖验收标准。
- 是否破坏现有 API、数据结构、权限、计费、缓存或历史记录。
- 错误处理是否友好且可追踪。
- 并发、重试、超时、幂等和边界输入是否合理。
- 是否引入密钥、个人路径、临时文件或调试输出。
- 测试是否覆盖核心路径和回归风险。
- 部署和回滚是否受影响。

## 专项检查矩阵

详细审查 rubric 见 `references/review-rubric.md`。当项目没有更具体规范时，以该 rubric 作为默认 review 标准。

| 类型 | 必查 |
| --- | --- |
| 前端 | 状态管理、表单校验、加载态、错误态、响应式、可访问性、console 错误 |
| 后端 API | 鉴权、参数校验、错误码、幂等、超时、日志、兼容性 |
| 数据库 | 迁移、索引、事务、回滚、旧数据兼容 |
| 异步任务 | 重试、幂等、状态流转、失败补偿、日志追踪 |
| AI/第三方 API | 降级、超时、费用、内容安全、返回格式变化 |
| 配置/部署 | env key、默认值、生产差异、重启需求、回滚 |
| skill 文档 | frontmatter、触发描述、流程完整性、脚本可执行性 |

## Review 方法

1. 先看总体 diff stat，判断是否有异常大改或生成产物。
2. 对每个入口追踪到核心逻辑和副作用。
3. 对新增分支构造失败路径和边界输入。
4. 对删除逻辑判断是否有替代路径。
5. 对测试文件判断是否真正会失败于旧 bug。
6. 对配置文件判断 dev/prod 是否一致。

## 输出格式

```markdown
**Review 结论**
- Pass / Needs Fix / Blocked

**发现项**
- P1 `文件:行` 问题、影响、建议

**测试缺口**
- 

**残余风险**
-
```

## 严重程度

- P0：会导致数据丢失、严重安全问题、生产不可用。
- P1：核心路径失败、明显回归、计费或权限错误。
- P2：边界问题、可恢复错误、缺少关键测试。
- P3：可维护性、命名、文档或小范围体验问题。

## 修复策略

- P0/P1：必须修复或让用户明确接受风险，不能静默放过。
- P2：能低风险修复就修；否则进入测试或发布风险列表。
- P3：只在不扩大范围时修复，避免无关重构。

## Pass 条件

- 没有未处理的 P0/P1。
- 验收标准有实现路径。
- 测试缺口已被补齐或明确记录。
- 部署风险已转交 `pola-deploy-release-gate`。

## Artifact 输出

阶段结束时输出：

```markdown
artifact: review
fields:
  status:
  findings:
  fixed:
  test_gaps:
  residual_risks:
```
