# Pola Devlog Writer · 项目开发日志撰写器

把"完成一次开发节点 → 写一段日志"这件事固化为可重复流程：固定 9 章节骨架 + 标准条目模版 + 一键统计脚本，确保日志结构稳定、数据有源、长期可读。

## 0. 何时启用

| 场景 | 调用方式 |
|------|---------|
| 项目从未有日志 → 首次生成 | "用 pola-devlog-writer 给这个项目建开发日志" |
| 完成一次开发节点（commit / merge）→ 增量追加 | "在开发日志顶部补一条今天的提交" |
| 仅刷新代码规模数据 | "重新跑一遍代码统计并更新 §9" |
| 大版本里程碑切换 | "新增 Stage N 条目并归档前面的待办" |

## 1. 输出位置

固定为 **项目根目录** `开发日志.md`（中文项目）或 `DEVLOG.md`（英文项目）。  
**不要**放到 `docs/`、`.qoder/`、子模块目录，避免被淹没。

## 2. 文档骨架（9 章节固定）

首次创建时复制 `templates/devlog-skeleton.md` 到项目根目录改名为 `开发日志.md`，再按下表填充：

| 章节 | 内容要点 |
|------|---------|
| §1 项目概览 | 项目名 / 仓库 / 线上地址 / 当前 HEAD / 日志最近更新 / 核心能力 5-8 条 |
| §2 技术栈 | 表格列出框架/语言/数据库/缓存/部署，附目录结构要点 |
| §3 环境配置 | 关键环境变量表 + 启动命令 |
| §4 开发阶段里程碑 | **倒序** Stage，每个 Stage 3-5 条要点 |
| §5 变更流水日志 | **倒序** commit，最新置顶；新增条目用 `templates/entry.md` |
| §6 待办与规划 | 🟢 已完成 / 🟡 进行中 / 🔵 规划 / 🔴 风险 四色清单 |
| §7 测试与文档索引 | 表格链接根目录所有 `.md` 测试报告与 PRD |
| §8 日志维护约定 | 复述模版，提醒每次更新需同步 §6、§9 |
| §9 代码规模统计 | 9.1 当前快照 / 9.2 阶段增量 / 9.3 演进观察 / 9.4 重新生成方法 |

> §9 的数据**必须**通过 `scripts/codestats.sh` 跑出来填，不要凭印象写。

## 3. 三种工作流

### 工作流 A · 首次创建（全新项目）

```
1. cp <skill-dir>/templates/devlog-skeleton.md ./开发日志.md
2. 跑 git log --pretty=tformat:'%h|%ad|%s' --date=short → 拿全部 commit
3. 按日期/语义分组成 Stage（§4 + §5）
4. 跑 bash <skill-dir>/scripts/codestats.sh [src_dir] → 填充 §9
5. 列出根目录所有 *.md 文档 → 填充 §7
6. §1 顶部填入：HEAD 短 hash + 当天日期
```

### 工作流 B · 增量追加一条（最常用）

```
1. git log -1 --shortstat HEAD → 拿到 hash / 标题 / +adds/-dels / 文件数
2. 用 templates/entry.md 在 §5 顶部插入新条目
3. 更新 §1「当前 HEAD」「日志最近更新」两个字段
4. 若涉及待办状态变化，同步勾选 §6
5. 若代码量变化 > 100 行，重跑 codestats.sh 刷新 §9
6. 不动其他章节
```

### 工作流 C · 仅刷新代码规模

```
1. bash <skill-dir>/scripts/codestats.sh [src_dir]
2. 把输出替换 §9.1 / §9.2 表格
3. 更新 §9 顶部"数据时间"与 HEAD hash
4. 不动其他章节
```

## 4. §5 变更流水条目模版

参见 `templates/entry.md`，核心字段：

```markdown
### [YYYY-MM-DD] `<短 hash>` <提交标题>

**目标**：一句话说明这次解决了什么问题。
**变更点**：
- 关键改动 1（带文件名/函数名）
- 关键改动 2
**影响文件**：<文件数> · +<add> / -<del>
**验证**：<本地 / 线上 / Browser Agent / 测试报告链接>
**关联**：<issue / PR / 需求 / 前一 commit>
```

**多 commit 集群**（同一天 ≥ 5 个 commit）改用表格形式：

```markdown
### [YYYY-MM-DD] <主题集群>（N 个 commit）

| hash | 主题 |
|------|------|
| `xxxxxxx` | … |
```

## 5. §6 待办四色规范

| 颜色 | 含义 | 用法 |
|------|------|------|
| 🟢 已完成 | 本 sprint 内完成、待归档 | 勾选后保留到下个版本切换时再删 |
| 🟡 进行中 / 可观察 | 正在做或需要监控 | 列原因，不写日期承诺 |
| 🔵 规划中 | 已立项未启动 | 按优先级排序 |
| 🔴 风险 / 技术债 | 不修会出事 | 必须有 owner 或缓解方案 |

## 6. 撰写语气与风格

- **第三人称**，不写"我"、"我们"。
- **结论前置**：每条变更点先说做了什么，再补背景。
- **链接代码**：引用文件用反引号包裹路径，如 `app/src/lib/rss/engine.ts`。
- **数字必带单位**：`+138 行`、`3MB → 30KB`、`9 个文件`。
- **不写营销话术**：禁用"全面提升"、"重磅升级"等空词；改用具体动词 + 数据。
- **每条 ≤ 8 行**：超长拆 sub-bullet 或拆多个 commit 条目。

## 7. 校验清单（提交前自检）

- [ ] §5 新条目在顶部，hash 用反引号包裹
- [ ] §1「当前 HEAD」「日志最近更新」已更新
- [ ] §9 数据时间与 §1 一致
- [ ] 所有内部锚点链接（`#X-章节`）可点击
- [ ] 没有时间敏感措辞（"昨天"、"本周内"）
- [ ] 文档总行数 < 500（超出考虑拆 `开发日志-archive.md`）

## 8. 工具脚本

### `scripts/codestats.sh`

**用途**：一次输出代码快照（按扩展名 + 按目录）和 git 阶段增量。

```bash
bash <skill-dir>/scripts/codestats.sh [src_dir] [scripts_dir]
# 默认 src_dir = app/src，scripts_dir = app/scripts
# 也可传入项目实际源码目录，例如：
bash <skill-dir>/scripts/codestats.sh src .
```

**输出三段**：
1. `=== 代码快照（按扩展名）===` — ts/tsx/js/py/css 等文件数与行数
2. `=== 代码快照（按目录）===` — 业务/库/组件/脚本各目录占比
3. `=== 阶段增量（按日期分组）===` — 每个开发节点的 commits / 文件次 / +adds/-dels
4. `=== 总计 ===` — 整个仓库历史累计

把输出粘到 §9.1 / §9.2 即可。

## 9. 反模式（不要这样做）

- ❌ 把 PRD / 需求文档塞进 §1 → 用 §7 链接外部文件
- ❌ 用 emoji 替代实际数据（"🚀🚀🚀 超大更新"）→ 写 `+579 行`
- ❌ 未验证就写"已上线" → 必须有 §5「验证」字段证据
- ❌ 把待办从 §6 复制到 §5 凑数 → 二者只在状态切换时联动
- ❌ 在 §5 顶部放未提交的 WIP → 只记 git 已落盘的 commit
- ❌ §9 数据手算 → 永远走 `scripts/codestats.sh`，避免误差

## 10. 完整示例

参见本仓库（pola-devlog-writer 起源项目）`PolaNews/开发日志.md`，覆盖：
- 9 章节完整填充
- 29 个 commit 倒序流水
- 5 个 Stage 阶段切片
- §9 代码规模统计三段式数据

复用时只需替换项目名、技术栈表、环境变量与 commit 内容即可。
