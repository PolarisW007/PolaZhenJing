# 智能排会助手（Wukong Meeting Assistant）

输入会议主题和参会人（支持 @人名、@群、@部门），自动查询所有人闲忙状态，基于空闲窗口算法推荐最优时间段，用户确认后一键创建日程、添加参会人、预订会议室、创建视频会议。

> 产品需求文档：`PRD.md`

## 何时使用

- 用户需要协调多人（尤其 5 人以上、跨部门）会议时间
- 用户用自然语言描述排会需求，如"帮我找下周 xxx 都有空的 1 小时"
- 用户需要一站式完成：查闲忙 → 选时段 → 建日程 → 加人 → 订会议室

## 依赖

- **dws CLI**：所有钉钉操作均通过 dws CLI 完成，需已完成 `dws auth login` 认证
- Python >= 3.9（纯标准库，无第三方依赖）
- 辅助脚本位于 `scripts/` 目录，Agent 可通过 `execute_shell` 调用以提高可靠性

### 脚本清单

| 脚本 | 职责 | CLI 用法 |
|------|------|---------|
| `scripts/dws_cli.py` | dws CLI 统一封装（执行/JSON解析/重试） | 仅作为库被其他脚本导入 |
| `scripts/time_utils.py` | 时间工具（ISO-8601/工作时间/slot生成） | 仅作为库被其他脚本导入 |
| `scripts/participant_resolver.py` | 参会人批量解析（@人名/@部门/@群 → userId） | `python participant_resolver.py --input '<JSON>' --include-self` |
| `scripts/free_slot_calc.py` | 空闲窗口算法（slot矩阵/评分/Top-N） | `python free_slot_calc.py --busy '<JSON>' --start ISO --end ISO --duration 60 --users '<JSON>' --top 3` |
| `scripts/meeting_creator.py` | 会议创建编排（日程+加人+会议室+视频会议） | `python meeting_creator.py --title "..." --start ISO --end ISO --users "uid1,uid2" --user-names "张三,李四" [--need-room] [--need-video]` |

> **Agent 可以选择直接调用 dws CLI 命令（适合简单场景），也可以调用脚本（适合复杂场景如多人解析、空闲计算）。两种方式等价。**

## 执行流程

**严格按以下 7 个步骤顺序执行。每步的 dws 命令必须加 `-f json` 以获取结构化输出，写操作加 `-y` 跳过确认。**

### Step 1：解析用户意图

从用户的自然语言输入中提取以下信息：

| 字段 | 说明 | 默认值 |
|------|------|--------|
| `topic` | 会议主题 | 若缺失，**追问用户** |
| `participants` | 参会人列表（人名/群名/部门名） | 若缺失，**追问用户** |
| `duration` | 会议时长（分钟） | 60 |
| `timeRange` | 搜索时间范围 | 未来 5 个工作日 09:00-18:00 |
| `needRoom` | 是否需要会议室 | false（用户提到"会议室"时为 true） |
| `needVideo` | 是否需要视频会议 | false（用户提到"视频会议"/"线上"时为 true） |

**自然语言时间解析规则：**

| 用户说法 | 解析为 |
|---------|--------|
| "下周" | 下周一 09:00 ~ 下周五 18:00 |
| "明天下午" | 明天 13:00 ~ 18:00 |
| "这周内" | 今天当前时间 ~ 本周五 18:00 |
| "尽快" | 今天剩余时间 + 未来 3 个工作日 |
| "下周二下午2-5点" | 下周二 14:00 ~ 17:00 |

如果关键信息（主题、参会人）缺失，**必须追问用户，不可假设**。

### Step 2：参会人解析与展开

根据参会人的类型，分别调用不同的 dws 命令将其解析为 userId 列表。

#### 2a. @人名 → userId

```bash
dws contact user search --query "{人名}" -f json
```

- 返回结果中取 `userId` 和 `name`
- **同名处理**：如果返回多个结果，展示列表（附部门信息）让用户选择
- **未找到**：提示"未找到'{人名}'，请确认姓名或花名"

#### 2b. @部门 → 展开全部成员

```bash
# 第一步：搜索部门，获取 deptId
dws contact dept search --query "{部门名}" -f json

# 第二步：获取部门成员列表
dws contact dept list-members --ids {deptId} -f json
```

- 如果部门搜索返回多个匹配，展示列表让用户选择
- 部门成员过多（>50人）时，提示用户确认是否继续

#### 2c. @群 → 展开全部成员

```bash
# 第一步：搜索群，获取 openConversationId
dws chat search --query "{群名}" -f json

# 第二步：获取群成员列表
dws chat group members --id {openConversationId} -f json
```

- 群成员支持分页（`--cursor`），首次从 0 开始
- 群成员过多（>50人）时，提示用户确认是否继续

#### 2d. 自动包含当前用户

```bash
dws contact user get-self -f json
```

将当前用户的 userId 加入参会人列表（如果尚未包含）。

#### 2e. 合并去重

将以上所有来源的 userId 合并，去重。记录每个 userId 对应的姓名，用于后续展示。

**完成后向用户确认参会人列表**，格式如：
```
👥 已解析参会人（共N人）：张三、李四、王五...
如有遗漏请补充，确认无误请说"继续"。
```

### Step 3：批量闲忙查询

将所有 userId 拼接为逗号分隔字符串，调用闲忙查询：

```bash
dws calendar busy search \
  --users "{userId1},{userId2},{userId3},..." \
  --start "{timeRange.start}" \
  --end "{timeRange.end}" \
  -f json
```

**注意事项：**
- `--start` 和 `--end` 必须为 ISO-8601 格式，如 `2026-04-14T09:00:00+08:00`
- 返回的是每个用户的**忙碌时段**列表，不是空闲时段
- 如果某用户查询失败（未开启日历/无权限），标记为"无法查询闲忙"，不阻塞其他人
- 忙碌时段数据为空的用户，视为该时间范围内全天空闲

### Step 4：空闲窗口计算与推荐

这是技能的核心算法步骤，由 Agent LLM 推理完成。

**算法流程：**

1. **构建时间槽矩阵**：将目标时间范围按 **30 分钟粒度**切分为 slot 数组。仅保留工作时间段（默认 09:00-12:00, 13:00-18:00，排除午休 12:00-13:00）。

2. **标记忙碌 slot**：遍历每个用户的忙碌时段，将被占用的 slot 标记为该用户不可用。

3. **寻找连续空闲窗口**：在 slot 矩阵中，找到连续 N 个 slot（N = 所需时长 ÷ 30）的窗口，计算每个窗口的可参加人数。

4. **评分排序**（按优先级从高到低）：
   - **全员可参加** > 部分人可参加
   - 同等可参加人数时：**工作时间（9:00-18:00）** > 非工作时间
   - 同等条件时：**上午 > 下午**
   - 同等条件时：**离当前时间越近越优先**

5. **输出 Top-3 推荐时段**，每个时段包含：
   - 开始时间、结束时间（格式：`周X MM/DD HH:MM - HH:MM`）
   - 可参加人数 / 总人数
   - 不可参加的人员姓名（如有）

**输出格式（必须严格遵循）：**

```
🗓 智能排会助手 — 推荐结果

📋 会议主题：{topic}
👥 参会人（共{N}人）：{name1}、{name2}、{name3}...
⏱ 所需时长：{duration} 分钟

推荐时段：
┌─────┬──────────────────────────┬──────────┬─────────────┐
│ 排名 │ 时段                      │ 可参加    │ 冲突人员     │
├─────┼──────────────────────────┼──────────┼─────────────┤
│ ⭐ 1 │ 周X MM/DD HH:MM - HH:MM │ N/N 全员  │ 无           │
│  2  │ 周X MM/DD HH:MM - HH:MM │ N-1/N   │ {name}      │
│  3  │ 周X MM/DD HH:MM - HH:MM │ N-1/N   │ {name}      │
└─────┴──────────────────────────┴──────────┴─────────────┘

请选择时段（输入 1/2/3），或说"换个时间"重新查找。
```

**异常处理：**
- **无共同空闲**：退化为"最多人可参加"的推荐模式，提示"N人在{时间范围}内无共同空闲，以下为最多人可参加的时段"，并建议放宽时间范围或减少必须参加人员
- **查询范围过大（>2周）**：提示用户缩小范围

### Step 5：用户确认选择

等待用户选择推荐时段。支持以下交互：

| 用户输入 | 行为 |
|---------|------|
| "1" / "2" / "3" 或 "第一个" / "选1" | 选择对应时段，进入 Step 6 |
| "换个时间" / "都不合适" | 追问新的时间范围约束，回到 Step 3 |
| "取消" / "算了" | 终止流程，不创建任何内容 |
| "加个会议室" | 将 `needRoom` 设为 true |
| "也约个视频会议" | 将 `needVideo` 设为 true |
| "能不能找个全员都有空的" | 调整算法只推荐全员可参加的时段 |

**必须等用户明确确认后才能进入创建步骤，绝不可自动创建。**

### Step 6：创建日程 + 添加参会人

用户确认时段后，按以下顺序执行：

#### 6a. 创建日程

```bash
dws calendar event create \
  --title "{topic}" \
  --start "{selectedSlot.start}" \
  --end "{selectedSlot.end}" \
  --desc "由智能排会助手自动创建\n参会人：{参会人姓名列表}" \
  -f json -y
```

- 从返回结果中提取 `eventId`，后续步骤依赖此 ID
- **创建失败**：重试 1 次，仍失败则提示用户手动创建并给出完整参数

#### 6b. 添加参会人

```bash
dws calendar participant add \
  --event {eventId} \
  --users "{userId1},{userId2},{userId3},..." \
  -y
```

- 参会人添加部分失败时，报告成功/失败列表，不回滚已创建的日程
- 添加成功后，参会人会自动收到钉钉日程邀请通知

#### 6c. 预订会议室（当 needRoom = true）

```bash
# 搜索空闲会议室
dws calendar room search \
  --start "{selectedSlot.start}" \
  --end "{selectedSlot.end}" \
  --available \
  -f json

# 从结果中选择合适的会议室（优先匹配容量需求）
# 预订会议室
dws calendar room add --event {eventId} --rooms {roomId} -y
```

- 如果无空闲会议室，提示用户
- 如果会议室被抢占（预订时已被占用），推荐其他空闲会议室
- 会议室容量选择：参会人数 ≤ 会议室容量 ≤ 参会人数 × 2（避免太大浪费）

#### 6d. 创建视频会议（当 needVideo = true）

```bash
dws conference meeting create \
  --title "{topic}" \
  --start "{selectedSlot.start}" \
  --end "{selectedSlot.end}" \
  -f json -y
```

- **重要**：视频会议不会自动关联日历日程。需要将会议链接信息追加到日程描述中
- 如果日程已创建，可通过 `dws calendar event update`（如可用）更新描述；否则在创建日程时预先在 `--desc` 中预留视频会议信息占位

### Step 7：输出完成摘要

全部操作完成后，输出以下格式的确认摘要：

```
✅ 会议已创建！

📋 {topic}
🕐 {日期} {星期} {开始时间} - {结束时间}
👥 参会人：{name1}、{name2}、{name3}...（已发送邀请）
🏢 会议室：{会议室名称}（{容量}人间）  ← 仅当预订了会议室时显示
🔗 视频会议：已创建（会议号附在日程描述中）  ← 仅当创建了视频会议时显示
```

## 全局规则

### CLI 调用规范

1. **所有 dws 命令必须加 `-f json`** 以获取结构化 JSON 输出，便于解析
2. **写操作（create/add）必须加 `-y`** 跳过交互确认（Agent 模式）
3. **时间格式统一使用 ISO-8601**，时区为 `+08:00`，如 `2026-04-15T14:00:00+08:00`
4. **userId 列表用逗号分隔**，无空格，如 `uid1,uid2,uid3`
5. 命令执行失败时，先检查 stderr 信息，重试 1 次后再报告用户

### 错误处理

| 错误场景 | 处理策略 |
|---------|---------|
| 用户搜索无结果 | 提示确认姓名/花名，建议换个关键词 |
| 部门/群搜索多个匹配 | 展示列表让用户选择 |
| 闲忙查询某用户失败 | 跳过该用户，标记"无法查询"，不阻塞流程 |
| 无共同空闲时段 | 退化推荐 + 建议放宽条件 |
| 日程创建失败 | 重试 1 次，仍失败给出手动创建指引 |
| 参会人添加部分失败 | 报告成功/失败列表 |
| 会议室被抢占 | 推荐其他空闲会议室 |

### 安全与权限

- 闲忙查询仅返回忙碌/空闲状态，不暴露具体事件内容（API 层已保证隐私）
- 日程以当前用户身份创建，参会人收到钉钉日程邀请通知
- **创建日程前必须经用户确认**，绝不自动执行

### 性能预期

| 指标 | 目标 |
|------|------|
| 参会人解析（10人以内） | < 5 秒 |
| 闲忙查询（20人 × 5天） | < 10 秒 |
| 端到端（输入到推荐） | < 30 秒 |
| 日程创建到完成 | < 10 秒 |

## 示例对话

### 示例 1：基础排会

**用户**：帮我约福锤和枫华下周的1小时会议，主题是悟空技能评审

**Agent 执行**：
1. 解析：topic="悟空技能评审", participants=["福锤","枫华"], duration=60, timeRange=下周一~五
2. `dws contact user search --query "福锤" -f json` → userId=026828
3. `dws contact user search --query "枫华" -f json` → userId=xxx
4. `dws contact user get-self -f json` → 确认自己已在列表中
5. `dws calendar busy search --users "026828,xxx" --start "2026-04-14T09:00:00+08:00" --end "2026-04-18T18:00:00+08:00" -f json`
6. 计算空闲窗口 → 输出 Top-3 推荐
7. 用户选择 → 创建日程 → 添加参会人 → 完成

### 示例 2：跨部门 + 会议室

**用户**：帮我约开放平台全部门下周二下午开个2小时的Q2复盘，要个会议室

**Agent 执行**：
1. 解析：topic="Q2复盘", participants=["开放平台"(部门)], duration=120, timeRange=下周二13:00~18:00, needRoom=true
2. `dws contact dept search --query "开放平台" -f json` → deptId
3. `dws contact dept list-members --ids {deptId} -f json` → userId 列表
4. 确认参会人列表 → 闲忙查询 → 推荐 → 用户确认
5. 创建日程 → 添加参会人 → 搜索空闲会议室 → 预订 → 完成

### 示例 3：群成员 + 视频会议

**用户**：在悟空核心群约个30分钟线上站会

**Agent 执行**：
1. 解析：topic="站会", participants=["悟空核心群"(群)], duration=30, needVideo=true
2. `dws chat search --query "悟空核心群" -f json` → openConversationId
3. `dws chat group members --id {convId} -f json` → userId 列表
4. 确认参会人 → 闲忙查询 → 推荐 → 用户确认
5. 创建日程 → 添加参会人 → 创建视频会议 → 完成
